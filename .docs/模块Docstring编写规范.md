# 模块元注释 (Docstring) 编写规范指导

## 1. 总体结构与风格

- **格式**：使用三重双引号 `"""` 包裹的多行文档字符串。
- **位置**：紧跟在模块/类/函数的定义之后，之前无空行。
- **风格**：内容应**严谨、精简、无歧义**，使用中文，遵循“总-分-详”的结构。
- **视觉分隔**：使用等号 (`===`) 或注释行 (`# ===`) 进行逻辑区块的视觉分隔，增强可读性。

## 2. 核心组成部分 (推荐顺序)

一个完整的模块级注释应包含以下几个部分：

### 2.1 模块标题与一句话概述
- **格式**：首行即模块名，第二行用简短的一句话说明模块的核心职责。
- **目的**：让读者在3秒内理解模块是做什么的。

### 2.2 详细描述
- **内容**：对模块功能、设计意图、解决的核心问题进行进一步阐述。
- **目的**：提供上下文，说明“为什么”需要这个模块。

### 2.3 模块构成/主要组件
- **格式**：使用列表或简要的标题说明形式。
- **内容**：列举模块中主要的类、函数或子模块，并为每个组件提供**一句话的功能说明**。
- **目的**：提供模块的“地图”，让读者快速定位到关键部分。

### 2.4 使用说明/典型工作流
- **格式**：使用步骤式、场景化的描述。
- **内容**：清晰地说明如何使用本模块完成一个典型任务。可以按“首先…然后…最后…”的逻辑串联。
- **目的**：提供一份“快速上手指南”，降低使用门槛。

### 2.5 重要提示/注意事项
- **内容**：包含**安全警告**、**性能关键点**、**常见的配置或使用陷阱**、**依赖关系说明**等。
- **目的**：避免用户因未知的“坑”而导致错误或安全漏洞。

### 2.6 元信息 (可选)
- **内容**：可包含`版本`、`作者`、`最后修改日期`、`参考链接`等信息。
- **目的**：便于维护和追溯。

## 3. 各类注释补充要点

### 3.1 类 (Class) 注释
- **概述**：说明这个类代表什么实体或承担什么职责。
- **属性**：简要说明主要属性的作用（尤其对于`dataclass`或`__init__`中的属性）。
- **方法**：可不在此处列出，除非有特别需要说明的公开方法模式。

### 3.2 函数/方法 (Function/Method) 注释
- **格式**：推荐使用 **Google风格** 或类似格式。
- **必须部分**：
    1.  **功能简述**：一句话说明函数作用。
    2.  **Args (参数)**：列出每个参数的名称、类型和简短描述。
    3.  **Returns (返回)**：说明返回值的类型和含义。
    4.  **Raises (异常)**：列出可能主动抛出的异常及其触发条件。
- **可选部分**：`Examples` (示例)，`Notes` (备注)。

## 4. 良好注释示例特征

- **准确**：描述与代码行为完全一致。
- **简洁**：避免冗长的故事叙述，直击要点。
- **有用**：提供代码本身不能一目了然的信息（如设计意图、业务逻辑、使用场景）。
- **主动**：使用主动语态和现在时态（例如“本模块提供…”而非“该模块被用于提供…”）。
- **无歧义**：避免使用“可能”、“也许”、“处理”等模糊词汇，对“错误”、“无效”、“过期”等状态给出明确定义。

## 5. 要避免的陷阱

1.  **不要**重复代码：避免写“将参数a和参数b相加”这样的注释。
2.  **不要**描述实现细节：注释应关注“接口”和“契约”，而非“实现”。避免“这里用一个for循环遍历列表”。
3.  **不要**留下过时信息：代码变更时，**必须同步更新注释**。
4.  **不要**在模块注释中写长篇的安装、配置教程，这些应归于独立的`README.md`。
